## 一、存储架构：Redis + SQLite 混合存储

### 1.1 设计动机

| 需求 | 纯 SQLite 方案的不足 | Redis + SQLite 如何解决 |
|------|----------------------|------------------------|
| 多轮对话状态保持 | SQLite 写磁盘有延迟，高频读写会话状态效率低 | Redis 内存级读写，毫秒级响应 |
| LangGraph State 跨节点共享 | State 仅在单次图执行中存活，会话结束即丢失 | Redis 将会话状态持久化到内存，跨请求共享 |
| 高频临时数据（检索缓存、中间结果） | SQLite 不适合存大量临时 JSON | Redis Hash / String 灵活存储，自带 TTL 过期 |
| 结构化业务数据持久化 | SQLite 擅长，无需改变 | SQLite 保持不变，负责长期数据 |
| 分布式 / 多实例扩展 | SQLite 单机文件锁 | Redis 可被多个 Agent 实例共享（未来扩展） |

### 1.2 职责划分

```
┌─────────────────────────────────────────────────────────┐
│                    Redis（内存层）                        │
│                                                         │
│  会话状态缓存 (session:{user_id})                        │
│    ├── conversation_phase    # 当前对话阶段              │
│    ├── intent_history        # 最近 N 条意图记录         │
│    ├── temp_recommendations  # 本轮推荐结果（未确认前）   │
│    ├── temp_conflicts        # 本轮冲突检测结果           │
│    ├── temp_resources        # 本轮资源检索结果           │
│    └── pending_actions       # 等待用户操作的动作队列     │
│                                                         │
│  多轮对话上下文 (chat:{user_id})                         │
│    ├── messages[]            # 最近 K 轮对话历史          │
│    └── context_summary       # 超出窗口的摘要             │
│                                                         │
│  RAG 检索缓存 (rag:{hash})                               │
│    └── 缓存已检索过的 Query → 结果映射，省重复 Embedding  │
│                                                         │
│  TTL 策略：                                              │
│    - 会话状态：30 分钟无操作自动过期                     │
│    - 对话历史：24 小时                                   │
│    - 检索缓存：1 小时                                    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                   SQLite（持久层）                        │
│                                                         │
│  users              用户档案（主修、年级、兴趣）          │
│  courses            课程信息（代码、名称、学分、难度）     │
│  user_timetable     用户课表（主修 + 已选辅修）           │
│  study_progress     学习进度（成绩、考勤、作业）          │
│  conflict_history   冲突检测历史记录                      │
│  resource_feedback  资源推荐反馈（用于优化推荐）          │
│  reports            已生成的学业报告存档                  │
│                                                         │
│  无 TTL，数据永久保存                                    │
└─────────────────────────────────────────────────────────┘
```

### 1.3 数据同步策略

```
写入流程:
  1. 先写 Redis（更新会话状态，用户立即看到反馈）
  2. 异步写 SQLite（通过队列或后台任务，保证数据最终一致）

读取流程:
  1. 先查 Redis（命中则直接返回，延迟 ~1ms）
  2. 未命中则查 SQLite（延迟 ~5-10ms）
  3. 查到的结果回填 Redis（热数据升温）

关键业务数据（课表变更、成绩录入）:
  双写 + 事务保护：Redis 写成功 → SQLite 写成功 → 才返回确认
```

## 二、LangGraph State 设计

### 2.1 State 定义（精简版，仅存流转必要信息）

```python
from typing import TypedDict, List, Optional, Literal
from enum import Enum

class ConversationPhase(str, Enum):
    IDLE = "idle"
    AWAIT_COURSE_SELECTION = "await_course_selection"      # 等待用户从推荐中选课
    AWAIT_PROGRESS_INPUT = "await_progress_input"           # 等待录入成绩/考勤
    AWAIT_CONFLICT_DECISION = "await_conflict_decision"     # 等待用户对冲突处理做决定
    AWAIT_RESOURCE_FEEDBACK = "await_resource_feedback"     # 等待用户反馈资源是否有用

class AgentState(TypedDict):
    # ── 身份标识 ──
    user_id: str
    session_id: str                     # 每次会话唯一 ID，对应 Redis Key

    # ── 路由相关 ──
    intent: str                         # 当前识别的意图枚举值
    conversation_phase: str             # ConversationPhase 当前阶段

    # ── 权限控制 ──
    has_minor_course: bool              # 是否已选辅修（从 Redis/SQLite 读取）
    has_major_timetable: bool           # 是否已录入主修课表

    # ── 当前请求的中间产出（不存历史全量！） ──
    current_recommendations: List[dict]  # 课程规划产出
    current_conflicts: List[dict]        # 冲突检测产出
    current_resources: List[dict]        # 资源推荐产出

    # ── 对话消息 ──
    messages: List[dict]                # 当前轮次的对话消息

    # ── 输出控制 ──
    final_response: str                 # 最终返回给用户的内容
```

### 2.2 Redis ↔ State 映射

```python
# Redis Key 设计
SESSION_KEY = "session:{user_id}:{session_id}"
CHAT_KEY    = "chat:{user_id}"

# 进入 LangGraph 节点时：从 Redis 加载 State
def load_state_from_redis(user_id: str, session_id: str) -> dict:
    """每个 Graph 节点执行前调用，从 Redis 恢复最新状态"""
    r = get_redis()
    state_data = r.hgetall(SESSION_KEY)
    if not state_data:
        # Redis 中没有 → 从 SQLite 冷启动
        state_data = load_from_sqlite(user_id)
        r.hset(SESSION_KEY, mapping=state_data)
        r.expire(SESSION_KEY, 1800)  # 30 分钟
    return state_data

# 离开 LangGraph 节点时：写回 Redis
def save_state_to_redis(state: AgentState):
    """每个 Graph 节点执行后调用，持久化状态变更"""
    r = get_redis()
    r.hset(SESSION_KEY, mapping={
        "intent": state["intent"],
        "conversation_phase": state["conversation_phase"],
        "has_minor_course": str(state["has_minor_course"]),
        "has_major_timetable": str(state["has_major_timetable"]),
        # 序列化列表字段
        "current_recommendations": json.dumps(state["current_recommendations"]),
        "current_conflicts": json.dumps(state["current_conflicts"]),
        "current_resources": json.dumps(state["current_resources"]),
    })
    r.expire(SESSION_KEY, 1800)
```

### 2.3 多轮对话状态流转示例

```
用户: "帮我推荐辅修课程"
  → conversation_phase = AWAIT_COURSE_SELECTION
  → Redis 记录: phase=AWAIT_COURSE_SELECTION, recos=[...]

用户: "我选第二个，金融方向"
  → 路由层先读 Redis: phase == AWAIT_COURSE_SELECTION
  → 不经过意图分类，直接路由到课程规划 Agent
  → Agent 从 Redis 取出 current_recommendations
  → 解析"第二个"，确认选课 → 写 SQLite → 更新 Redis phase=IDLE

用户: "帮我检测课程冲突"
  → 路由层读 Redis: phase == IDLE
  → 关键词匹配 → intent_check_conflict
  → 冲突检测 Agent ...
```

## 四、路由 Agent 详细设计

### 4.1 意图枚举

```python
from enum import Enum

class Intent(str, Enum):
    # 业务意图
    PLAN_COURSE       = "plan_course"       # 辅修课程规划与推荐
    TRACK_PROGRESS    = "track_progress"     # 查询/录入学习进度
    CHECK_CONFLICT    = "check_conflict"     # 检测时间冲突
    RECOMMEND_RES     = "recommend_res"      # 推荐学习资源
    GEN_REPORT        = "gen_report"         # 生成学业报告
    # 操作意图
    DATA_MANAGE       = "data_manage"        # 数据管理（录入课表、修改信息）
    COURSE_ADJUST     = "course_adjust"      # 课程调整（退课、换课）
    # 非业务意图
    CHAT_QA           = "chat_qa"            # 闲聊/学科问答
```

### 4.2 路由流程

```python
def route_user_input(user_input: str, state: AgentState) -> str:
    """
    返回目标节点名称，供 LangGraph 使用
    """

    # ── 第 0 步：检查是否处于等待阶段 ──
    phase = state.get("conversation_phase")
    if phase == ConversationPhase.AWAIT_COURSE_SELECTION:
        return "course_planner_agent"
    if phase == ConversationPhase.AWAIT_PROGRESS_INPUT:
        return "progress_tracker_agent"
    if phase == ConversationPhase.AWAIT_CONFLICT_DECISION:
        return "conflict_checker_agent"
    if phase == ConversationPhase.AWAIT_RESOURCE_FEEDBACK:
        return "resource_recommender_agent"

    # ── 第 1 步：关键词粗筛 ──
    intent = keyword_match(user_input)
    if intent:
        return intent_to_node(intent)

    # ── 第 2 步：LLM 精判 ──
    intent = llm_classify(user_input, state)
    return intent_to_node(intent)


def keyword_match(text: str) -> Optional[str]:
    """关键词匹配，命中则直接返回意图，无需调用 LLM"""
    rules = [
        (["推荐辅修", "选辅修", "规划课程", "辅修推荐", "辅修方案"], Intent.PLAN_COURSE),
        (["考勤", "成绩", "作业", "查看进度", "进度查询", "录入成绩", "录入考勤"], Intent.TRACK_PROGRESS),
        (["时间冲突", "撞课", "课表冲突", "考试冲突", "检测冲突"], Intent.CHECK_CONFLICT),
        (["资料", "习题", "课件", "学习资源", "网课", "推荐资源", "找资料"], Intent.RECOMMEND_RES),
        (["生成报告", "总结情况", "学业报告", "出报告", "分析报告"], Intent.GEN_REPORT),
        (["修改课表", "录入课表", "添加课程", "更新信息", "修改专业"], Intent.DATA_MANAGE),
        (["退课", "换课", "退选", "调整课程"], Intent.COURSE_ADJUST),
    ]
    for keywords, intent in rules:
        if any(kw in text for kw in keywords):
            return intent
    return None


def intent_to_node(intent: Intent) -> str:
    """意图 → LangGraph 节点名"""
    mapping = {
        Intent.PLAN_COURSE:    "course_planner_agent",
        Intent.TRACK_PROGRESS: "progress_tracker_agent",
        Intent.CHECK_CONFLICT: "conflict_gate",          # 先经过权限门
        Intent.RECOMMEND_RES:  "resource_recommender_agent",
        Intent.GEN_REPORT:     "report_generator_agent",
        Intent.DATA_MANAGE:    "data_manager_agent",
        Intent.COURSE_ADJUST:  "course_adjuster_agent",
        Intent.CHAT_QA:        "chat_qa_handler",
    }
    return mapping[intent]


def llm_classify(text: str, state: AgentState) -> Intent:
    """LLM 精判兜底，用于关键词未命中的情况"""
    prompt = f"""
    你是意图分类器。根据用户输入判断意图，只返回枚举值。

    用户信息：主修={state.get('major')}, 已选辅修={state.get('has_minor_course')}

    可选意图：
    - plan_course：推荐辅修、规划课程
    - track_progress：查询或录入学习进度、成绩、考勤
    - check_conflict：检测课程/考试时间冲突
    - recommend_res：推荐学习资料
    - gen_report：生成学业报告
    - data_manage：管理个人信息或课表数据
    - course_adjust：退课或换课
    - chat_qa：闲聊、学科知识问答（与辅修业务无关）

    用户输入：{text}
    只输出意图枚举值：
    """
    response = call_llm(prompt)
    try:
        return Intent(response.strip())
    except ValueError:
        return Intent.CHAT_QA  # 安全兜底
```

### 4.3 权限拦截节点（conflict_gate）

```python
def conflict_gate(state: AgentState) -> str:
    """
    冲突检测的准入条件检查。
    这个节点不调用 LLM，纯规则判断。
    """
    if not state.get("has_major_timetable"):
        state["final_response"] = (
            "您还未录入主修课表，无法进行冲突检测。\n"
            "请先录入主修课程信息（课程名 + 上课时间 + 考试时间），"
            "可以输入「录入课表」开始。"
        )
        return "end"  # 直接结束，不进入冲突检测

    if not state.get("has_minor_course"):
        state["final_response"] = (
            "您暂未选择辅修课程，无需进行冲突检测。\n"
            "可以输入「推荐辅修」让我帮您规划辅修方案。"
        )
        return "end"

    # 通过准入检查，进入真正的冲突检测
    return "conflict_checker_agent"
```

## 五、五个业务 Agent 详细设计

### 5.1 课程规划 Agent

**职责**：根据用户主修专业、空闲时间、兴趣方向，推荐辅修专业 + 课程组合

```
输入：
  - user_id → 从 SQLite 读取用户档案（major, available_slots, interests）
  - 若 conversation_phase == AWAIT_COURSE_SELECTION：
      从 Redis 读取 current_recommendations，解析用户的序号选择

处理流程：
  1. 【新推荐场景】
     a. 构造 RAG 查询 → 检索知识库（辅修专业简介 + 课程信息 + 报名规则）
     b. LLM 整合检索结果，生成 Top-3 辅修方案
     c. 方案存入 Redis current_recommendations
     d. 更新 Redis conversation_phase = AWAIT_COURSE_SELECTION

  2. 【用户选择场景】
     a. 从 Redis 取出 current_recommendations
     b. 解析用户输入（"第2个"、"选金融"）
     c. 将选定的课程写入 user_timetable（course_type='minor'）
     d. 更新 SQLite users.has_minor = 1, users.minor_program = 选定的专业
     e. 更新 Redis has_minor_course = True, phase = IDLE
     f. 提示用户：可以检测冲突、查看资源

知识库结构（新增 MD 文件）：

### 5.2 进度跟踪 Agent

**职责**：展示学习进度、接收手动/模拟数据录入

```
输入：
  - user_id
  - 用户查询意图（"查看进度" 或 "录入成绩 85"）

处理流程：
  1. 【查询进度】
     a. 从 SQLite 读取 study_progress（按 course_code 分组）
     b. 表格化展示：课程名 | 周次 | 考勤 | 作业 | 小测 | 状态
     c. 计算：总出勤率、平均分、落后课程标记
     d. 结果回填 Redis（缓存 5 分钟）

  2. 【录入进度】
     a. 解析用户输入 → 提取 course_code、week、score、attendance
     b. INSERT 到 SQLite study_progress

### 5.3 冲突检测 Agent

**职责**：自动识别主修/辅修上课、考试时间冲突，给出调课/取舍建议

```
输入：
  - user_id → 从 SQLite 读取 user_timetable
  - has_minor_course（已由 gate 检查）

处理流程：
  1. 从 SQLite 读取所有 course_type='major' 和 course_type='minor' 的课程
  2. 时间区间重叠判断：

     def has_overlap(course_a, course_b):
         """精确到节次的重叠检测"""
         # 周次重叠
         week_overlap = max(a.week_start, b.week_start) <= min(a.week_end, b.week_end)
         # 星期重叠
         day_overlap = a.day_of_week == b.day_of_week
         # 节次重叠
         period_overlap = max(a.period_start, b.period_start) < min(a.period_end, b.period_end)
         return week_overlap and day_overlap and period_overlap

  3. 区分冲突类型：
     - exam_overlap：两台考试时间重叠 → severity=critical
     - course_overlap：两门课时间重叠且都是必修 → severity=warning
     - course_overlap：一门必修 + 一门可选 → severity=info

  4. 对每个冲突，调用 RAG 检索辅修规则库 → LLM 生成 ≥2 个解决方案
  5. 结果写入 Redis current_conflicts + SQLite conflict_history
  6. 更新 phase = AWAIT_CONFLICT_DECISION（如果冲突需要用户决定）

输出示例：
  🔍 课程冲突检测报告

  │ CS301 数据结构（主修）📅 2025-12-20 14:00-16:00      │

### 5.4 资源推荐 Agent

**职责**：基于当前学习进度，RAG 检索课件、习题、网课、讲座等资料

```
输入：
  - user_id → 从 SQLite 读取：已选课程、study_progress 中的薄弱点

处理流程：
  1. 【识别薄弱点】
     - 从 study_progress 中找出 homework_score < 70 的课程
     - 从 study_progress 中找出 quiz_score < 60 的课程
     - 标记为 "薄弱课程 + 薄弱周次对应的知识点"

  2. 【分级检索】
     a. 精确检索：course_code → 该课程全部资料（课件、习题、笔记）
     b. 语义检索：薄弱知识点 → Vector DB 中相似内容 → 跨课程相关章节
     c. 外部检索：调取预索引的网课链接、公开课资源

  3. 【LLM 整理】
     - 去重：同一份资料可能被多种检索方式命中
     - 排序：按相关度 + 难度匹配度 + 用户偏好（视频/文字）排序

  4. 结果写入 Redis current_resources（缓存到 session 结束）

  5. RAG 检索缓存：同一 query 在 1 小时内复用 Redis 缓存结果

## 六、LangGraph 工作流编排

### 6.1 完整状态图

```python
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.redis import RedisCheckpointer  # 或自定义 RedisSaver

# ── 构建图 ──
workflow = StateGraph(AgentState)

# 注册节点
workflow.add_node("router", router_node)                     # 路由
workflow.add_node("course_planner_agent", course_planner)    # 课程规划
workflow.add_node("progress_tracker_agent", progress_tracker)# 进度跟踪
workflow.add_node("conflict_gate", conflict_gate)            # 权限门
workflow.add_node("conflict_checker_agent", conflict_checker)# 冲突检测
workflow.add_node("resource_recommender_agent", resource_rec) # 资源推荐
workflow.add_node("report_generator_agent", report_gen)      # 报告生成
workflow.add_node("data_manager_agent", data_manager)        # 数据管理
workflow.add_node("course_adjuster_agent", course_adjuster)  # 课程调整
workflow.add_node("chat_qa_handler", chat_qa)                # 兜底对话

# 入口
workflow.set_entry_point("router")

# 路由分发
workflow.add_conditional_edges(
    "router",
    lambda state: state["intent"],
    {
        "plan_course":      "course_planner_agent",
        "track_progress":   "progress_tracker_agent",
        "check_conflict":   "conflict_gate",
        "recommend_res":    "resource_recommender_agent",
        "gen_report":       "report_generator_agent",
        "data_manage":      "data_manager_agent",
        "course_adjust":    "course_adjuster_agent",
        "chat_qa":          "chat_qa_handler",
    }
)

# 权限门分发
workflow.add_conditional_edges(
    "conflict_gate",
    lambda state: "conflict_checker_agent" if state.get("_gate_pass") else END,
    {
        "conflict_checker_agent": "conflict_checker_agent",
        END: END,
    }
)

# 所有业务节点执行完毕后 → END
for node in ["course_planner_agent", "progress_tracker_agent",
             "conflict_checker_agent", "resource_recommender_agent",
             "report_generator_agent", "data_manager_agent",
             "course_adjuster_agent", "chat_qa_handler"]:
    workflow.add_edge(node, END)

# ── 编译（挂载 Redis Checkpointer 实现状态持久化） ──
redis_checkpointer = RedisCheckpointer(
    redis_url="redis://localhost:6379",
    ttl=1800  # 30 分钟
)
app = workflow.compile(checkpointer=redis_checkpointer)
```

## 七、完整数据流示例（端到端）

```
场景：用户从零开始，推荐辅修 → 选课 → 检测冲突 → 生成报告

Step 1: 用户输入「推荐辅修」
  Router:
    → phase=IDLE → 关键词命中 "推荐辅修"
    → intent=plan_course
  Course Planner:
    → 从 SQLite 读取用户档案
    → RAG 检索辅修知识库
    → LLM 生成 3 个方案
    → 写入 Redis: current_recommendations, phase=AWAIT_COURSE_SELECTION
    → 返回：推荐方案列表

Step 2: 用户输入「选方案一」
  Router:
    → phase=AWAIT_COURSE_SELECTION → 直接路由到 course_planner
  Course Planner:
    → 从 Redis 读取 current_recommendations
    → 确认方案一 → 写入 SQLite: user_timetable (3 门辅修课)
    → 更新 SQLite: users.has_minor=1, minor_program="金融辅修"
    → 更新 Redis: has_minor_course=True, phase=IDLE
    → 清除 Redis: current_recommendations
    → 返回：选课成功 + 提示可检测冲突

Step 3: 用户输入「检测冲突」
  Router:
    → phase=IDLE → 关键词命中 → intent=check_conflict
  Conflict Gate:
    → Redis: has_minor_course=True, has_major_timetable=True
    → 通过 ✓
  Conflict Checker:
    → 从 SQLite 读取所有课表
    → 时间重叠算法 → 发现 1 个考试冲突 + 1 个上课冲突
    → RAG 检索辅修规则 → LLM 生成解决方案
    → 写入 Redis: current_conflicts, phase=AWAIT_CONFLICT_DECISION
    → 写入 SQLite: conflict_history
    → 返回：冲突报告 + 方案选择

Step 4: 用户选择「方案A：申请缓考」
  Router:
    → phase=AWAIT_CONFLICT_DECISION → 直接路由
  Conflict Checker:
    → 更新 SQLite: conflict_history.user_decision
    → 更新 Redis: phase=IDLE
    → 清除 Redis: current_conflicts

Step 5: 用户输入「生成报告」
  Router:
    → 关键词命中 → intent=gen_report
  Report Generator:
    → 从 Redis 读取: current_conflicts(null), current_resources(null)
    → 从 SQLite 读取: users, study_progress, conflict_history, reports(上期)
    → 数据缺失兜底：冲突已解决但有历史记录，资源未检索
    → 趋势对比（vs 上期）
    → LLM 生成报告 Markdown
    → 写入 SQLite: reports
    → 清除 Redis 本轮临时数据
    → 返回：完整报告 + 可下载 .md 文件
```

## 十、关键风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Redis 宕机导致会话状态丢失 | 用户需重新对话 | Redis 持久化（RDB/AOF）+ SQLite 作为冷启动回源 |
| LLM 输出格式不稳定 | 解析失败，流程中断 | 所有 LLM 调用加结构化输出约束 + try/except 兜底 |
| RAG 检索质量差 | 推荐资料不相关 | 人工标注种子数据 + 检索结果 LLM 重排 + 用户反馈微调 |
| 时间冲突算法边界情况 | 漏判/误判冲突 | 单元测试覆盖所有重叠模式（完全重叠/部分重叠/包含/不重叠） |
| 知识库内容不足 | 推荐方案单一 | 初期覆盖核心辅修专业（3-5 个），后续按需扩展 |

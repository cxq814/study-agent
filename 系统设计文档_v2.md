## 一、系统概述

### 1.1 项目定位

面向大学生的辅修学习全生命周期管理助手，覆盖**选课规划 → 进度跟踪 → 冲突检测 → 资源推荐 → 报告生成**五大环节，基于 LangGraph 多智能体协作框架实现。

### 1.2 技术栈

| 层次 | 技术选型 | 说明 |
|------|---------|------|
| 编排框架 | LangGraph StateGraph | DAG 拓扑，条件边分发 |
| 向量检索 | ChromaDB (PersistentClient) | 知识库语义检索 |
| 嵌入模型 | BGE-small-zh-v1.5 / TF-IDF 回退 | 国内网络自动降级 |
| 会话存储 | Redis + 内存字典 fallback | 30min TTL，无 Redis 自动降级 |
| 业务存储 | SQLite (WAL 模式) | 6 表，外键约束 |
| LLM | Anthropic API（可选） | 无 API key 时全量回退规则模板 |
| CLI | Python 原生 REPL | UTF-8 终端 |

```
IDLE ─────────────────────────────────────────────> 正常意图分类
  │
  ├─ "推荐辅修" → AWAIT_COURSE_SELECTION ──> 等待选课
  │                  ├─ "选方案一" → IDLE（选课完成）
  │                  ├─ "重新推荐" → AWAIT_NEW_INTERESTS
  │                  │     └─ "金融" → AWAIT_COURSE_SELECTION
  │                  └─ "重新推荐+法学" → AWAIT_COURSE_SELECTION（直接）
  │
  ├─ "录入成绩" → AWAIT_PROGRESS_INPUT ──> 等待结构化数据
  │
  ├─ "录入课表"/"修改信息" → AWAIT_DATA_INPUT ──> 等待录入课表/编辑个人信息
  │
  ├─ "检测冲突" → [conflict_gate] → AWAIT_CONFLICT_DECISION
  │                  ├─ "忽略" → IDLE
  │                  ├─ "调课" → IDLE（清空辅修）
  │                  └─ 全局指令 → IDLE（跳回路由）
  │
  └─ "推荐资源" → AWAIT_RESOURCE_FEEDBACK
                     ├─ "有帮助" → IDLE
                     ├─ "换一批" → 重新检索
                     └─ 全局指令 → IDLE（跳回路由）
```

### 2.3 数据存储架构

```
┌─────────────────────────────────────────────────────┐
│                    Redis (会话层)                     │
│  session:{uid}:{sid} → Hash (phase, recs, conflicts) │
│  rag_cache:{md5}     → String (TTL 1h)               │
│  chat:{uid}          → List (最近 20 条)              │
│  内存 fallback: _memory_fallback dict                │
├─────────────────────────────────────────────────────┤
│                   SQLite (业务层)                     │
│  users, courses, user_timetable,                    │
│  study_progress, conflict_history, reports          │
└─────────────────────────────────────────────────────┘
```

- **Redis 存临时态**：会话状态、对话历史、RAG 缓存
- **SQLite 存持久态**：用户档案、课表、学习进度、冲突记录、报告
- **内存兜底**：Redis 不可用时自动切换内存字典，进程内多轮对话不丢状态

## 三、Phase 4 设计方案

### 3.1 新增 Agent 决策

| Agent | 是否新增 | 理由 |
|-------|---------|------|
| **课程调整 Agent** (course_adjuster) | ✅ 是 | 退课/换课是独立业务场景，需多步确认流程（选课→确认→写库），不能混入冲突检测的快捷清空 |
| **数据管理 Agent** (data_manager) | ✅ 是 | 用户需管理主修课表、个人信息、兴趣标签，与课程调整（策略决策）不同，偏向 CRUD |
| **报告生成 Agent** (report_generator) | ✅ 替换 stub | 聚合 4 个数据源（进度+冲突+资源+课表），生成结构化报告 |
| **兜底问答 Handler** (chat_qa) | ✅ 替换 stub | 当前所有未匹配输入都掉进 echo stub，用户体感差。对接 LLM + RAG 做学科问答 |

**为什么不合并课程调整和数据管理？**
- `course_adjust` 触发词是"退课/换课/改选"，是**策略决策**，需要展示当前课表→用户选择→确认→写库的多步交互
- `data_manage` 触发词是"录入课表/修改信息/我的课表"，是**数据维护**，偏向表单式编辑
- 两者触发词、交互模式、状态流转都不同，分开更清晰

### 3.3 Phase 4 Agent 详细设计

#### 3.3.1 报告生成 Agent（report_generator）

```
触发词: "生成报告", "学业报告", "总结情况", "出报告", "综合报告"

流程:
  1. 聚合数据:
     - get_timetable(user_id) → 课表概览
     - get_study_progress(user_id) → calc_progress_trend() → 趋势图数据
     - get_conflict_history(user_id) → calc_conflict_summary() → 冲突统计
     - get_user(user_id) → 基本信息
  2. 风险量化:
     - calc_risk_score(progress, conflicts) → 综合风险分
  3. 生成报告:
     - LLM: 将所有数据喂给 LLM 生成结构化报告（优先）
     - 模板: Markdown 模板填充（回退）
  4. 导出:
     - export_markdown() → data/exports/report_YYYYMMDD.md
  5. 持久化:
     - add_report(user_id, report_type, content_md, summary_json)
  6. 输出:
     - 报告摘要 + 导出路径 + 操作建议

多轮: 不需要。报告生成是一次性操作，phase 保持 IDLE。
```

#### 3.3.2 数据管理 Agent（data_manager）

```
触发词: "录入课表", "修改信息", "我的课表", "添加课程", "更新信息"

支持的子功能:
  a. 查看课表: 展示当前全部课程（主修+辅修），按星期排列
  b. 录入主修课表: 逐门添加（课程名+时间+地点+考试时间）
  c. 修改个人信息: 更新年级、可用时段、兴趣标签
  d. 删除课表条目: 按序号选择删除

流程（以"录入主修课表"为例）:
  1. 询问课程名 → AWAIT_DATA_INPUT
  2. 用户输入 "CS101 周一1-3节 教三楼201"
  3. 解析 → add_timetable_entry() → 确认 → 询问是否继续
  4. 用户输入 "完成" → IDLE

注: 需要新增 ConversationPhase: AWAIT_DATA_INPUT（或复用现有机制）
```

#### 3.3.3 课程调整 Agent（course_adjuster）

```
触发词: "退课", "换课", "退选", "改选", "取消辅修"

流程:
  1. 展示当前辅修课表
  2. 用户选择要退的课程: "退 FIN201"
  3. 确认: "确认退选 FIN201 公司金融？(y/n)"
     - AWAIT_COURSE_ADJUSTMENT 等待确认
  4. 确认后:
     - delete_timetable_entry(entry_id)
     - 重算学分
     - 如果退的是最后一门辅修课 → 自动 has_minor=0
  5. 如果是换课: "换 FIN201 换 DS201"
     - swap_course(user_id, "FIN201", "DS201")
     - validate_schedule() 预检新冲突
     - 如有新冲突 → 展示冲突 → 询问是否继续

与冲突检测中"调课"的关系（系统内存在两套课程调整功能）:
  - 冲突检测快捷清空（conflict_checker._handle_decision）:
    触发词「调课/调整/换时间/改时间」，一键删除全部辅修数据，重新来过。
    Router 通过 adjust_schedule 意图 → 直达 conflict_checker_agent（跳过 gate）。
  - 课程调整 Agent（course_adjuster，Phase 4）:
    触发词「退课/换课/改选/取消辅修」，逐门退课/换课，带确认流程和冲突预检。
    Router 通过 course_adjust 意图 → course_adjuster_agent。
  - 两者触发关键词不同（"调课" ≠ "退课"）、流程独立、互不冲突。
```

#### 3.3.4 兜底问答 Handler（chat_qa）

```
触发: 所有未匹配到其他意图的输入（包括闲聊、学科问题）

实现:
  1. 如果是学科问题（含关键词如"什么是/解释/为什么/怎么做"）:
     - RAG 检索知识库 → 拼接相关上下文
     - LLM 生成回答（或模板拼接）
  2. 如果是闲聊:
     - LLM 自由回答（或预设回复列表）
  3. 如果完全无法处理:
     - 友好提示 + 引导到可用功能

注: 这是系统的"最后一道防线"，当前 echo stub 导致用户输入无反馈。
```

## 八、已知局限与待改进项

| 序号 | 问题 | 影响 | 建议 |
|------|------|------|------|
| 1 | Embedding 模型在国内网络几乎必定回退 TF-IDF | RAG 检索无语义相似度 | Phase 5 支持本地模型缓存 + HF 镜像配置 |
| 2 | ChromaDB 索引无增量更新 | 新增知识库文件需 force_rebuild | Phase 5 增加文件 MD5 去重 |
| 3 | settings.CHROMA_PERSIST_DIR 与 indexer.py 实际路径不一致 | 配置无效 | 即时修复：统一为 settings 中的路径 |
| 4 | CoursePlanner 的 _default_day/period/location 硬编码 | 新增课程无法自动分配时间 | Phase 4 从知识库 YAML frontmatter 读取 |
| 5 | RAG Retriever 的类型过滤用 Python 后过滤而非 ChromaDB where | 多取 2x-3x 数据再丢弃，浪费 | Phase 5 改为 ChromaDB metadata where |
| 6 | sqlite_models.py 的 Pydantic 模型未被使用 | 无校验层 | **保留文件**，Phase 4 工具层将调用 Pydantic 模型做全局数据入参校验 |
| 7 | chat_qa_handler 是 echo stub | 所有未匹配输入无意义回复 | Phase 4 实现兜底问答 |
| 8 | LLM API key 默认空，全系统依赖模板回退 | 推荐质量受限 | 文档说明 + Phase 5 前端配置入口 |
